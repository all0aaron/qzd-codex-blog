为什么这样设计

用户同时推进多个项目，每个项目只需打开 index.html 就能看到该项目的一切：需求文档、UI 预览入口、开发进度、测试结果、启动方式。不需要翻多个文件，查看效率最高。

index.html 总控台规范
项目最重要的文件，必须做好

必须包含的章节

  • 项目概览：项目名称、技术栈、当前进度一句话描述
  • 阶段进度看板：9 个阶段各有状态徽章（待开始 / 进行中 / 已完成）
  • 产品需求：完整功能模块列表，含日期标注
  • UI 预览区：所有 ui_*.html 的卡片入口，点击跳转
  • 数据库设计：数据表结构说明
  • API 文档：所有接口的请求/响应说明
  • 测试报告：接口测试和功能测试结果
  • 项目总结：启动命令、默认账号、遇到的问题
  • 部署说明：打包命令、部署方式

视觉风格要求

  • 左侧固定导航栏（深色背景）+ 右侧内容区（白色背景），与本文件同款布局效果
  • 专业、简洁、现代感强，像真正的项目文档平台
  • 纯 HTML + CSS + 少量 JS，不依赖构建工具，双击直接打开即可查看
  • UI 预览区用卡片网格展示，每张卡片有页面名称，点击跳转对应 HTML
  • 阶段进度用颜色区分：灰色=待开始，蓝色=进行中，绿色=已完成

更新机制

每推进一步，立刻更新 index.html 对应章节，不等全部完成再更新。用户随时打开 index.html 都能看到当前最新进度。

💡 所有 ui_xxx.html 页面顶部必须有「← 返回总控台」按钮，底部有上一页 / 下一页导航，形成完整的预览体验。

阶段0：需求分析与项目规划
收到大方向后的第一步

收到用户的项目大方向后，立即自主完成以下分析，然后创建 index.html 骨架并写入分析结果。

需要分析的维度

  • 用户角色：有哪些角色（管理员/普通用户/访客/商家等），各角色的权限边界是什么
  • 功能模块：至少 5 个核心模块，每个模块下 3~5 个具体功能点，按 P0/P1/P2 排优先级
  • 涉及的端：Web 后台、Web 前台、小程序、App、官网，各端需要哪些页面
  • 第三方依赖：是否需要微信登录、支付、短信验证码、地图、文件上传等
  • 数据规模：大概需要建多少张表，主要存哪些数据
  • 技术方向：根据项目类型，自己评估用什么前端框架、什么后端框架最合适，写进 index.html 的技术方案章节
⚙️ 技术选型完全由你自主决定。不同项目用不同技术是正常的。你在阶段0就要确定好本项目的技术方案并说明原因，写进 index.html。

阶段1：产品设计
把需求整理成结构清晰的产品文档

将阶段0的分析结果整理成完整的产品文档，全部写入 index.html 的"产品需求"章节。

必须包含的内容

  • 项目背景：这个项目解决了什么问题，面向哪类用户
  • 功能模块列表：每个模块列出具体功能点，标注优先级 P0/P1/P2 和所属端
  • 核心业务流程：主要流程的文字描述（注册流程、下单流程、审核流程等）
  • 页面清单：所有需要开发的页面名称、路由、所属端
  • 技术方案说明：选择了哪些技术，为什么这样选
⏱️ 这部分写得越详细越好，后续 UI 设计和开发都依赖这里的内容。每个需求点必须有日期标注。

阶段3：前端开发
用户确认 UI 后开始，技术自主选型
🧠 前端用什么框架、什么技术栈，由你根据项目类型自主判断，不要照搬固定模板。
例如：后台管理系统、用户前台、小程序、App、官网，每种类型适合的技术是不同的。你在阶段0就应该想好，这里按你定的方案实现。

开发要点（与具体技术无关，所有项目通用）

  • 路由配置完整，有权限守卫（未登录自动跳转登录页）
  • 所有接口调用统一封装，自动携带 token，统一处理错误提示
  • 表单有完整的前端验证逻辑（必填、格式、长度等）
  • 列表页有分页、搜索、筛选功能
  • 操作按钮有加载状态，成功/失败有明确提示
  • 适配移动端，至少不乱版，最好有完整响应式
  • 接口地址通过环境变量配置，不硬编码

阶段4：后端开发
技术自主选型，实现所有接口
🧠 后端用什么语言、什么框架，由你根据项目类型自主判断。
常规 Web 后台推荐 Java，轻量工具类项目可以考虑其他方案，具体由你决定并在 index.html 说明原因。

开发要点（所有项目通用）

  • 所有接口统一返回格式：{ code, msg, data }
  • 有全局异常处理，任何错误都不能直接返回 500 页面
  • 接口遵循 RESTful 风格（GET 查询、POST 新增、PUT 修改、DELETE 删除）
  • 需要登录的接口统一鉴权（JWT 或 Session，由你决定）
  • 有跨域配置，允许前端项目访问
  • 后端端口固定 8080，不要随意修改
  • 有 API 文档（接口列表、请求参数、响应格式），写入 index.html

阶段5：数据库设计
统一使用 MySQL，每项目独立建库
⚠️ 所有项目统一使用 MySQL，不使用其他任何数据库。
每个新项目必须新建独立的数据库，绝对不能修改或污染其他已有数据库中的任何表或数据！

数据库连接信息见下方「数据库固定配置」章节。

新项目数据库命名规则

命名格式：项目英文名_db，例如 blog_db、shop_db、task_db

建表规范

  • 每张表必须有：id（主键自增）、create_time、update_time、is_deleted（逻辑删除）
  • 每个字段必须有中文注释（COMMENT）
  • 字段名使用下划线命名（snake_case）
  • 不使用外键约束，用业务逻辑保证关联
  • 字符集统一：utf8mb4，排序规则：utf8mb4_unicode_ci

输出文件

  • database/init.sql：完整建表 SQL（含字段注释和初始化数据）
  • database/mock_data.sql：至少 10 条测试数据，供开发调试使用

把数据库表结构说明同步写入 index.html 的"数据库设计"章节。

Ralph 自动循环——AI 不中断持续开发
让 Codex 自己跑完所有任务，无需盯着
✅ Ralph 已集成到本项目。启动文件：启动Ralph-博客开发.bat（双击即可）。AI 会自动循环完成 prd.json 中的所有任务，中途断了重新双击继续，已完成的任务不会重复做。

Ralph 是什么

Ralph 是一个自主 AI 任务循环脚本，原理很简单：

  • 你把所有要做的功能写进 prd.json（任务清单）
  • Ralph 每次让 Codex 完成一个任务，完成后自动 git commit
  • 在 prd.json 中把该任务标记为完成（passes: true）
  • 自动继续下一个，直到所有任务完成
  • 你不需要盯着，去做别的事，回来看结果就行

关键文件说明

文件作用
prd.json任务清单，每个任务有 passes 字段记录完成状态
ralph.ps1主循环脚本（PowerShell），驱动 codex exec 反复运行
CODEX.md给 Codex 的指令模板，告诉它每次该做什么

Ralph 执行流程详解
每一次迭代发生了什么
STEP 1
读取任务
从 prd.json 找 priority 最小且 passes=false 的任务
STEP 2
构建提示词
把 CODEX.md + prd.json + progress.txt 合并成完整提示词
STEP 3
调用 Codex
执行 codex exec --dangerously-bypass-approvals-and-sandbox
STEP 4
Codex 工作
Codex 读文件、写代码、git commit，自动完成任务
STEP 5
标记完成
Codex 把 prd.json 中该任务的 passes 改为 true
STEP 6
循环
回到 STEP 1，直到所有任务完成或输出 COMPLETE

30个项目同步开发——通用模板方案
每个项目只需放入同一套文件，用一句话描述驱动 AI 自动展开
✅ 核心思路：Ralph 支持「Bootstrap 自动生成」—— 如果项目目录里没有 prd.json，只需有一个 项目说明.txt（一句话描述），Ralph 会先让 Codex 读取 序章.html 的规范，自动生成完整的 prd.json 任务清单，然后继续自动开发。

每个项目需要放的文件（全部一样，只有一个文件不同）

文件是否相同说明
序章.html✅ 所有项目完全一样AI 开发规范，不需要修改
ralph.ps1✅ 所有项目完全一样自动循环脚本（带 Bootstrap）
CODEX.md✅ 所有项目完全一样给 Codex 的执行指令模板
启动Ralph.bat✅ 所有项目完全一样双击启动，含 GitHub 自动化
项目说明.txt⭐ 每个项目不同一句话描述要开发什么，这是唯一需要改的文件

项目说明.txt 格式（一句话）

开发一个电商平台，有前台商品列表、商品详情、购物车、结算，后台支持商品CRUD、订单管理、用户管理。

开发一个任务管理工具，支持任务分类、优先级、截止日期，有团队协作和进度看板功能。

开发一个外卖小程序，有商家入驻、菜品展示、购物车下单、骑手接单配送流程和后台统计报表。

GitHub 自动化——自动建仓库、自动推送
启动Ralph.bat 已内置 GitHub 自动化逻辑
⚠️ 前提条件：需要安装并登录 gh（GitHub CLI）。安装后执行 gh auth login 完成授权，之后一劳永逸。

GitHub CLI 安装

// 方法1：官网下载
https://cli.github.com/

// 方法2：winget 安装
winget install --id GitHub.cli

// 登录（只需执行一次）
gh auth login

启动Ralph.bat 的 GitHub 自动化逻辑

检测条件自动执行的操作
没有 .git 目录自动执行 git init + 初始提交
没有 origin 远端自动用文件夹名创建 GitHub 私有仓库，设置 remote 并推送
Ralph 循环完成后自动 git push origin HEAD 推送所有提交

手动创建 GitHub 仓库（备用）

// 在项目目录执行：
git init -b main
git add .
git commit -m "init: project setup"
gh repo create 项目名称 --private --source=. --remote=origin --push
✅ 最终效果：双击 BAT → AI 自动开发 → 每个任务自动 commit → 全部完成后自动 push 到 GitHub。30个项目分别运行，每个都有独立的 GitHub 仓库，代码安全备份在云端。
